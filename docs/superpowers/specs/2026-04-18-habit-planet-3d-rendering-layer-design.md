# Habit Planet 3D Rendering Layer Technical Design Doc

## 1. 背景与目标

《习惯星球》项目需要在移动端 WebView 环境中，以 Three.js 渲染一颗会随着用户打卡天数 `checkInDays` 持续进化的 3D 星球。当前 3D 层的职责边界明确如下：

- 仅负责 3D 场景渲染、阶段演进、资产加载、镜头与灯光表现、特效控制与调试走查能力。
- 不负责页面 UI、网络请求、业务接口、埋点系统与业务状态管理。
- 业务侧前端或客户端通过状态管理或桥接，把真实 `checkInDays` 传递给 3D 渲染层。

本设计文档的目标是为 3D 渲染层定义一套适合比赛项目、但具备清晰边界与可实现性的技术方案，使系统在保守性能档的移动端 WebView 中稳定运行，并且优先保证阶段辨识度与整体构图气质。

## 2. 输入约束与设计前提

### 2.1 真实输入约束

- 场景完全由 `checkInDays` 驱动。
- 线上真实数据变化模式为连续递增，即主要场景为 `N -> N+1`。
- 当前本地已存在隐藏 Debug 入口，可模拟 `checkInDays +1` 以便独立开发和走查。

### 2.2 视觉演进模式

场景采用混合模式：

- 基础生态类内容长期累积保留。
- 里程碑资产按阶段执行升级替换。
- 高成本氛围特效定义为可降级能力。

### 2.3 视觉优先级

项目的视觉优先级按以下顺序执行：

1. 阶段辨识度
2. 整体构图气质
3. 核心英雄资产
4. 氛围特效

这意味着在性能受限、资产不足或时间不足时，优先保阶段主轮廓、机位、灯光和关键标志物，而非优先保留高成本特效。

### 2.4 设备与性能前提

- 目标设备档位为保守档，即优先兼容中低端 Android WebView。
- 阶段 5 和阶段 6 的特效允许在低端机自动降级。
- 当前不设定刚性的全局帧率承诺，但要求稳定优先，不允许因高阶段特效导致明显黑屏、长时间卡死或内存溢出。

### 2.5 资产供给前提

资产供给为混合供给：

- 既可能拿到完整 `GLB/GLTF` 成品。
- 也可能拿到通用模型配合独立贴图或材质。
- 资源组织方案必须同时支持两类接入方式。

### 2.6 镜头与过渡前提

- 正式方案使用半固定机位。
- 用户允许小范围拖拽观察，但不能破坏阶段的主视觉构图。
- 动画策略采用分层型：平时递增更新克制，跨阶段边界时再触发轻量升级演出。

## 3. 六阶段视觉规则

### 3.1 阶段定义

- 阶段 1，`1-3` 天：基础地表材质、幼苗、小石头。
- 阶段 2，`4-10` 天：增加灌木、小树、篝火与暖色点光。
- 阶段 3，`11-21` 天：丰富植被、加入土路与木屋骨架。
- 阶段 4，`22-45` 天：木屋完整、窗户发光、风车旋转、兔子入场、能量光轨开启。
- 阶段 5，`46-90` 天：植被更密、加入长椅和秋千、小动物出现，特效允许降级。
- 阶段 6，`91+` 天：终极升级，生命树替换主树，开启祝贺光环、星尘或光粒子，低端机允许简化实现。

### 3.2 参考图定位

六张阶段图片作为高保真视觉参考，主要用于定义：

- 每阶段的主视觉气质。
- 每阶段最需要被用户识别的关键元素。
- 哪些资产属于长期保留，哪些资产属于升级替换。

这些参考图不是逐像素还原目标。若受制于模型质量、贴图质量、包体、性能或实现周期，允许采用低成本近似方案，但必须保持阶段辨识度和整体构图方向一致。

## 4. 非目标

本方案明确不覆盖以下内容：

- 不负责业务 UI、页面交互流程与接口请求。
- 不负责企业级调试后台、远程日志平台和完整性能监控面板。
- 不追求参考图级别的逐像素精致度。
- 不要求所有特效在所有设备上强一致开启。
- 不在本阶段引入过重的通用引擎框架或高度配置化平台。

## 5. 整体架构设计

### 5.1 推荐架构

推荐使用“渲染内核 + 分层场景编排器 + 资产注册表 + 轻量 Debug 面板”的结构：

- `PlanetRenderer`
  - 管理 `scene`、`camera`、`renderer`、`controls`、`resize`、`raf` 与 `dispose`。
  - 不直接负责每个阶段摆放什么模型。
- `Stage Orchestrator`
  - 接收 `checkInDays`。
  - 计算当前阶段、阶段内进度、需要激活的层、替换的英雄资产、当前质量档位。
  - 生成过渡计划并驱动各层更新。
- `Asset Registry`
  - 统一管理模型、贴图、材质、实例模板、预加载与降级映射。
- `Layer Controllers`
  - 分层管理场景内容，避免 `Stage1 ~ Stage6` 文件持续膨胀。
- `Debug Panel State`
  - 仅用于开发与验收走查，不承担业务状态管理。

### 5.2 分层场景结构

建议按职责拆成稳定层：

- `TerrainLayer`：星球本体、基础地表、土路、草地覆盖。
- `VegetationLayer`：幼苗、花草、灌木、小树、主树。
- `StructureLayer`：木屋骨架、完整木屋、长椅、秋千、风车等建筑与设施。
- `CharacterLayer`：兔子、鹿、小鸟、蝴蝶等角色与生物。
- `FxLayer`：篝火发光、窗户发光、能量光轨、简单粒子。
- `FinaleLayer`：阶段 6 的祝贺光环、生命树增强表现、星尘与最终升级效果。

### 5.3 为什么不继续纯阶段类扩展

当前基于 `Stage1.ts` 到 `Stage6.ts` 的写法适合快速起步，但对以下问题不够友好：

- 混合模式下的累积保留与升级替换同时存在。
- 高阶段特效需要按质量档位降级。
- 阶段切换需要镜头、灯光与新资产协同过渡。
- Debug 跳阶段需要快速收敛到目标态，而非重放全部中间过程。

因此，阶段概念应保留，但运行时实现应升级为分层编排，而不是继续把所有逻辑塞进单阶段类中。

## 6. 场景状态机设计

### 6.1 双层状态

状态机建议分为两层：

- `Progress State`
  - 记录真实输入，如 `checkInDays`、当前阶段编号、阶段内进度。
- `Presentation State`
  - 记录表现态，如镜头参数、灯光参数、已激活层、当前质量档位、正在播放的过渡动画。

这样可以避免业务值直接硬切场景，让渲染层拥有自己的过渡缓冲。

### 6.2 关键快照结构

每次更新由 `Stage Orchestrator` 计算 `StageSnapshot`：

- `dayCount`
- `stageIndex`
- `stageProgress`
- `activeLayers`
- `heroAssets`
- `fxLevel`
- `cameraPreset`
- `lightingPreset`

再通过比较新旧快照生成 `TransitionPlan`。

### 6.3 三类更新路径

- `日常递增更新`
  - 线上主路径，主要是 `N -> N+1`。
  - 仅更新阶段内参数，不做重型演出。
- `关键阶段切换`
  - 在 `3 -> 4`、`10 -> 11`、`21 -> 22`、`45 -> 46`、`90 -> 91` 时触发。
  - 触发轻量升级演出。
- `调试强制跳转`
  - 仅用于 Debug 走查。
  - 目标是快速构建目标阶段，而不是从头回放全部历史。

### 6.4 资产分类规则

状态机内需明确三类资产：

- `Cumulative`
  - 长期保留，如基础草地、花草、基础树木、土路。
- `Upgradeable`
  - 阶段升级替换，如木屋骨架到完整木屋，普通主树到生命树。
- `Optional FX`
  - 可降级效果，如光轨、星尘、祝贺光环。

## 7. 资产加载策略与资源生命周期

### 7.1 资产分层

建议将资源分为以下层级：

- `L0 基础常驻资产`
  - 星球基础几何、基础材质、基础天空与基础灯光。
- `L1 阶段公共复用资产`
  - 草地、石头、花草、小树、通用材质与通用贴图。
- `L2 当前阶段英雄资产`
  - 篝火、木屋、风车、兔子、生命树等。
- `L3 下一阶段预加载资产`
  - 当前阶段稳定后只预加载下一阶段关键资源。
- `L4 弹性特效资产`
  - 粒子、光轨、祝贺光环、增强发光效果。

### 7.2 加载原则

- 不做全阶段一次性加载。
- 保证当前阶段最低可见版本优先出现。
- 首屏成功后再分批补齐细节与下一阶段关键资源。
- Debug 跳阶段同样遵循“先最低可见，再补细节”的顺序。

### 7.3 混合供给接入原则

- 完整 `GLB/GLTF` 优先作为英雄资产使用。
- 通用模型加独立贴图优先放在公共复用层。
- 同类资源尽量通过共享几何、共享材质和实例化复用。

### 7.4 资产取舍与替代清单

为了避免“现有模型质量差且重复堆叠导致性能问题”，资源选型必须遵循以下取舍规则：

- 大面积生态效果优先视为渲染表现问题，而不是模型堆叠问题。
- 草被、花海、地表覆盖、能量光轨、祝贺光环、星尘粒子优先使用材质层、贴图混合、实例化 patch 或程序化特效。
- 英雄资产才值得单独找模型或做精细资源接入。
- 现有模型若同时满足“效果差、面数高、节点深、难复用”，应直接淘汰，而不是勉强接入。

资源按实现策略分为四类：

- `必须找模型`
  - `hut-full`
  - `windmill`
  - `rabbit`
  - `life-tree`
- `优先自己拼低模或用基础几何体替代`
  - `campfire`
  - `hut-skeleton`
  - `bench`
  - `swing`
- `优先复用现有模型，但仅允许少量点缀或实例化复用`
  - `rock`
  - `bush`
  - `stylized_tree`
  - `flower`
- `不建议靠模型解决，改为材质层或程序化表达`
  - `grass cover`
  - `energy ring`
  - `halo`
  - `stardust`

针对草被这类最容易导致性能问题的视觉需求，明确采用以下三层表达：

- `远景层`：使用地表颜色、覆盖遮罩和材质混合表达整体变绿。
- `中景层`：使用少量草簇 patch，并通过 `InstancedMesh` 布局。
- `近景层`：只在英雄机位前景区域摆少量高质量草簇或花簇。

禁止把草被设计成“单个草模型复制几百份”的方案，因为这会显著增加 draw call、节点数量与 CPU 摆放成本，并且并不一定带来更好的效果。

### 7.5 生命周期策略

- `L0` 和核心 `L1` 常驻，不频繁释放。
- 混合模式中可累积的资产只更新参数与可见性。
- 替换型资产进入延迟回收队列，避免即时销毁导致卡顿。
- 一次性特效严格短生命周期，结束后立即回收。
- Debug 模式提供场景强制清理并重建能力。

## 8. 过渡动画与镜头灯光策略

### 8.1 过渡风格

采用分层型过渡：

- 日常 `N -> N+1` 更新克制，仅表现阶段内细微生长。
- 跨阶段边界时触发 `600ms - 1200ms` 轻量升级演出。

### 8.2 日常递增更新

阶段内部使用可插值参数驱动增长，例如：

- 草地覆盖率、花草密度、小树数量逐步提高。
- 土路清晰度、木屋骨架完成度逐步提高。
- 窗户发光强度、风车转速、能量光轨亮度按阶段内进度上升。

### 8.3 关键阶段演出

跨阶段时的推荐顺序：

1. 保持旧阶段主轮廓可见。
2. 镜头轻微纠偏到该阶段英雄机位附近。
3. 新资产以淡入、缩放浮现或轻微位移的方式出现。
4. 灯光参数缓慢过渡。
5. 如果设备允许，再叠加短时 highlight 或轻量特效。

### 8.4 半固定机位

相机策略定义如下：

- 每阶段维护一个 `Hero Camera Preset`。
- 用户只允许在小范围内 orbit。
- 系统在关键阶段切换时将镜头缓慢收敛回最佳构图附近。

表现层至少维护：

- `baseAzimuth`
- `basePolar`
- `distance`
- `targetOffset`
- `userOrbitDelta`

### 8.5 灯光策略

灯光不按阶段整套重建，而是维持稳定结构并按阶段改参数：

- 基础环境光
- 主方向光
- 氛围补光
- 局部重点光源，如篝火点光、窗户发光、生命树增强发光

这种方式比完全重建灯光更稳定，更适合移动端 WebView。

## 9. 移动端 WebView 性能优化规范

### 9.1 总体原则

- 性能预算优先服务于阶段辨识度和整体构图。
- 高阶段特效默认视为可选增强项。
- 所有效果必须存在可简化或可关闭版本。

### 9.2 几何与实例化

- 石头、花草、灌木、小树优先使用 `InstancedMesh` 或共享几何。
- 避免大量重复独立 Mesh 导致 draw call 急剧增加。
- 小装饰不追求单对象独立控制，优先保证整体密度感。
- 草被禁止用数百个独立 `GLTF` 草模型拼装，必须改用材质覆盖加少量实例化 patch。

### 9.3 材质与贴图

- 控制材质种类数量，减少 shader 组合爆炸。
- 大面积基础材质优先中低分辨率贴图。
- 高分辨率贴图只保给核心英雄资产。
- 发光、透明和粒子材质只用于少量关键对象。

### 9.4 灯光与阴影

- 保持少灯光、强氛围。
- 阴影不作为默认硬需求，如果开启需按质量档严格受控。
- 通过材质颜色、贴图与 emissive 近似替代部分昂贵光照效果。

### 9.5 特效降级规则

针对阶段 5 和阶段 6，运行时允许按质量档执行：

- 减少粒子数量。
- 降低动画更新频率。
- 关闭次要透明叠加或高频发光。
- 把持续飞行动画降级为低频巡航或静态摆件。
- 把复杂光轨降级为简单环形光带。

### 9.6 质量档位

定义轻量 `QualityTier`：

- `Tier 0`：低端保底
- `Tier 1`：默认均衡
- `Tier 2`：增强显示

判定可以结合：

- 首屏初始化耗时
- 最近若干帧平均帧时
- 当前资源加载压力

无论降级到哪个档位，都不允许牺牲阶段主轮廓与关键标志物存在性。

## 10. Debug 模式与 3D 视觉走查工具

### 10.1 定位

Debug 工具定位为默认隐藏的轻量 3D 视觉走查面板，服务对象主要是：

- 3D 前端开发者
- 客户端联调同学

它不是企业级调试后台，而是开发与验收辅助工具。

### 10.2 最小功能集合

- 手动输入 `checkInDays`
- 阶段快捷跳转
- `+1 / -1` 微调
- 重播当前阶段入场过渡
- 显示当前阶段、当前天数、当前质量档位
- 显示当前启用的主资产与特效级别
- 提供场景强制清理并重建入口

### 10.3 两种工作模式

- `跟随真实数据`
  - 只读观察真实业务值。
- `本地覆盖模式`
  - 临时覆盖 `checkInDays`，仅作用于本地 3D 预览。

推荐默认进入跟随真实数据模式，避免联调时误覆盖真实输入。

### 10.4 不纳入本期范围

- 远程日志平台
- 完整性能火焰图
- 参数全量可视化编辑器
- 权限体系与多人协作后台

## 11. 验收标准

### 11.1 整体验收

整体验收通过条件如下：

- 六个阶段在视觉上可明确分辨。
- 英雄机位下的构图气质与参考图方向一致。
- 每个关键阶段至少有 1 到 2 个核心标志物被稳定呈现。
- 高阶段特效允许降级，但降级后仍保有终局升级感。
- 首屏不黑屏，阶段切换不出现明显长时间卡死。
- Debug 走查可稳定切阶段、重播过渡、观察质量档位。

### 11.2 分阶段验收

- 阶段 1：能明显看出星球从荒芜进入萌芽状态。
- 阶段 2：能明显看出篝火和早期生态建立。
- 阶段 3：能明显看出生活痕迹出现，土路和木屋骨架形成家园雏形。
- 阶段 4：能明显看出家园成形，风车、兔子、光轨形成里程碑升级。
- 阶段 5：能明显看出繁荣感增强，生活设施与生态密度提升。
- 阶段 6：能明显看出终局升级，生命树或终极主资产具有强辨识度。

## 12. 风险与应对

### 12.1 主要风险

- 资产质量、命名、贴图规格与坐标系不统一。
- 阶段逻辑继续堆积在单阶段类中，导致高阶段维护困难。
- 参考图过于精致，实际资源无法完全等价实现。
- WebView 环境差异导致性能与图形表现不稳定。
- Debug 面板功能蔓延，侵占主视觉实现时间。

### 12.2 应对策略

- 所有资产统一走 `Asset Registry` 描述接入。
- 所有阶段逻辑统一收敛到 `Stage Orchestrator + Layer Controllers`。
- 明确参考图是视觉目标而非逐像素还原标准。
- 运行时内建 `QualityTier`，高阶段特效天然支持降级。
- Debug 面板只保留走查必需功能，坚持轻量化边界。

## 13. 现有代码迁移建议

结合当前代码结构，建议分阶段演进：

### 13.1 保留的部分

- 组件层继续保持轻量，当前 `PlanetCanvas` 的职责边界正确。
- 状态输入链路可继续沿用现有 `growth store -> PlanetCanvas -> renderer` 模式。

### 13.2 需要收敛的部分

- `planetRenderer.ts` 应从“渲染宿主 + 场景细节堆叠”收敛成“渲染宿主 + 编排器入口”。
- `Stage1.ts` 到 `Stage6.ts` 不应继续扩展为大而全的阶段脚本，应逐步抽离为层规则和阶段配置。

### 13.3 推荐迁移顺序

1. 先引入 `Stage Orchestrator`，保持现有渲染结构可运行。
2. 再把可复用内容抽到分层控制器。
3. 然后引入 `Asset Registry` 与简单 `QualityTier`。
4. 最后补充 Debug 走查面板与高阶段特效降级。

## 14. 总结

本方案在比赛项目的现实约束下，选择分层场景编排型架构，重点解决：

- 六阶段混合模式下的累积保留与升级替换。
- 移动端 WebView 中的资产加载、资源生命周期与性能降级。
- 以半固定机位和轻量升级演出保证阶段辨识度与构图气质。
- 以轻量 Debug 走查工具支撑开发、联调与验收。

在优先级 `阶段辨识度 > 整体构图气质 > 核心英雄资产 > 氛围特效` 的前提下，该设计能以较低的系统复杂度，换取更稳定的开发推进路径与更清晰的验收标准，是当前《习惯星球》3D 渲染层最合适的技术方案。
